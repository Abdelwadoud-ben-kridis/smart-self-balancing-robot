Mechanical CAD, electronics architecture, and firmware design are finalised. This document records the Project Review Board sign-off for the transition from validated design into physical prototype: print, wire, flash, and tune.

Objective

A two-wheeled self-balancing robot with articulated legs. The robot transitions cleanly between sitting (66 mm) and standing (220 mm) under active balance, then holds vertical pitch while accepting drive commands from a vision-driven behaviour layer.

Validates the mechanical, electrical, and firmware design against the success criteria below and authorises the build phase. Re-review is triggered if assembled mass exceeds 600 g, the PID loop fails to stabilise within two tuning sessions, or any rail draws over its regulator headroom.

Mirrored four-bar linkages with equal 97 mm arms.

Why equal arms

With two equal 97 mm links, the upper-arm and lower-arm angles always sum to a constant relative to gravity — the foot tracks essentially straight up and down through the travel range. The CAD walk-through confirms ≤ 4 mm of forward drift at full extension.

03
Fig. 03 · Electronics

Dual-controller architecture separated by latency budget.

The ESP32 handles all time-critical work: MPU-6500 sensing, PCA9685 servo output, and L298N motor drive within a deterministic 10 ms cycle. The Raspberry Pi 5 runs the vision pipeline and behaviour state machine, communicating with the ESP32 over a single UART link at 115200 baud.

Fig. 04 · Power

Four regulated rails from a single 3S 18650 pack.

Drive motors run directly off the pack to isolate large transient currents from the regulated rails. Servos and the Raspberry Pi 5 each have a dedicated buck converter; the ESP32 and 3.3 V logic share the 5 V rail via the ESP32's onboard LDO.

Fig. 06 · Software

Complementary filter feeding a discrete PID at 100 Hz.

Arduino C++ firmware on the ESP32. The complementary filter weights gyro integration over short timescales against the accelerometer reference over long ones; the fused pitch drives a discrete PID controller, with a minimal serial protocol exposing runtime commands and live gain tuning.

control.ino
// complementary filter — pitch fusion pitch = 0.98 * (pitch + gyroY * dt) + 0.02 * atan2(accelX, accelZ); // discrete PID float err = target - pitch; integral += err * dt; float deriv = (err - prevErr) / dt; float u = Kp*err + Ki*integral + Kd*deriv; prevErr = err; // drive — clamp and write u = constrain(u, -255, 255); driveMotors(u); // stand-up interpolation (called at 100Hz) if (state == STANDING) { step++; float t = step / 200.0f; // 200 × 10ms = 2 s knee = lerp(80, 0, ease(t)); hip = lerp(-80, 0, ease(t)); writeServos(knee, hip); if (step == 100) pidEnabled = true; // 50% }
07
Fig. 07 · Risks

Five tracked risks, all with designed-in mitigations.

Mechanical, electrical, and firmware risks identified during design review. Each carries a concrete, already-implemented mitigation. Select any row to expand.

R-01 Medium Servo torque insufficient if body too heavy Mech +
Mitigation — torque verified with 2.4× safety margin at the 471 g target body mass. Use PLA or PETG, 18650 cells (not SLA), and cap infill at 30% on non-load-bearing parts. Re-run the calculation if assembled mass exceeds 600 g.
R-02 High PID instability / oscillation on first boot FW +
Mitigation — serial tuning interface is built in. Start with Kp = 5 and step up by +5 per trial. When oscillation appears, back off 20% and hold. Only after Kp and Kd are stable should Ki be introduced. Always tether the robot during these sessions.
R-03 High Robot falls during stand / sit transition Mech + FW +
Mitigation — stand-up runs as a 2 second smooth interpolation. The PID balance loop activates at the 50% point of the sequence, giving the legs time to reach a stable geometry before active balancing engages. Tested on carpet, with a spotter on first runs.
R-04 Medium Boot-time motor glitch from strapping GPIO EE +
Mitigation — ESP32 GPIO 12, 14, and 15 are strapping pins with known boot-time behaviour. GPIO 12 is unused. GPIO 14 (ENC1_B) defaults LOW at boot — safe. GPIO 15 (H_IN4) defaults HIGH — mitigated by driving it LOW in the first line of setup() before any motor code runs. Pin map documented in the firmware header.
R-05 Low Battery over-discharge / cell damage Power +
Mitigation — BMS board included in the design and wired in series with the pack. Under-voltage cutoff protects the 18650 cells; for the first few tuning sessions, also use a separate battery alarm as a redundant audible warning. Never bypass the BMS.

Fig. 10 · Architecture

Two processors, four bounded queues, one UART link.

The ESP32 runs a deterministic 100 Hz balance loop driven by a minimal serial protocol. The Pi 5 runs a multi-threaded vision pipeline with bounded inter-thread queues; producers are non-blocking and stale data is dropped at the queue boundary.

Fig. 3 · Pi 5 thread + ESP32 dataflow

→ Pi 5 to ESP32

← ESP32 to Pi 5
T:pitch,motor_l,motor_r,state\n // 20 Hz telemetry stream // state ∈ {SIT, STD, BAL} // |pitch| > 45° → Pi 5 sends P:SIT

Processes
Plumbing
ESP32 · real-time balance loop
100 Hz · 6 steps per cycle
  1. I/OSample inertial measurement unit10 ms tick

    Read MPU-6500 over SPI; pull raw gyro Y and accel X/Z from registers.

  2. FusionComplementary angle filter~25 μs

    pitch = 0.98 * (pitch + gyroY * dt) + 0.02 * atan2(accelX, accelZ) — 98% gyro short-term, 2% accel long-term.

  3. ControlDiscrete PID compute~10 μs

    u = Kp·e + Ki·∫e + Kd·ė where e = target − pitch. Integrator + derivative held across ticks.

  4. DriveMotor output to L298NPWM

    Clamp u ∈ [−255, +255]; sign flips direction pin; magnitude written as 8-bit PWM duty.

  5. ServoStance interpolation2 s / 200 steps

    On STAND/SIT command, ease knee + hip angles over 200 × 10 ms steps; PID enables at step 100 (50%).

  6. CommsSerial I/Oasync

    Parse incoming M: / P: / G: / E: commands; broadcast T: telemetry every 50 ms (20 Hz).

Raspberry Pi 5 · vision pipeline threads
4 workers · non-blocking
  1. SourceCameraThread30 fps

    Grab frame from USB webcam; push to frame_queue(maxsize=2); drop oldest if full so the camera never blocks.

  2. InferenceCVPipeline~20 fps

    Pull frame; run YOLOv8n for persons + MediaPipe for hands and pose; assemble SceneState; push to scene_queue(maxsize=2).

  3. LogicBehaviorManager10 Hz

    Pull SceneState; evaluate state machine; decide drive command; push to command_queue(maxsize=10).

  4. TransportSerialComm115200 baud

    Drain command_queue and write to Serial2 (GPIO 35 RX / GPIO 2 TX); read telemetry lines back from ESP32; push to telemetry_queue(maxsize=20).

Behaviour state machine · high-level autonomy
6 states · 10 Hz tick
  1. IdleIDLEdefault

    Robot seated; only motion-detection runs on the webcam; waits for a person to enter the frame or a manual command.

  2. TransitionSTANDING2 s

    Person detected or peace-sign gesture received; sends P:STAND; waits 2 s; transitions into FOLLOWING.

  3. ActiveFOLLOWING10 Hz

    Tracks person centroid; computes turn_error + distance_error; emits M: commands at 10 Hz.

  4. OverrideGESTUREpriority

    Open palm → E-STOP. Fist → SIT → IDLE. Peace sign → STAND (from IDLE only). Gesture state preempts all others.

  5. AutonomousPATROLno-person 30 s

    If no person is seen for 30 s, the robot navigates SLAM waypoints and sweeps the environment.

  6. AlertALERTon unknown face

    Unknown face detected; Telegram notification dispatched; robot holds position until a known face is observed.

Person follower · closed-loop pursuit
640×480 · 10 Hz
  1. DetectYOLOv8n bounding box~20 fps

    Detector outputs a person bbox [x1, y1, x2, y2] in a 640 × 480 frame. Highest-confidence person retained.

  2. CentroidHorizontal centreO(1)

    bbox_cx = (x1 + x2) / 2; compared against frame centre frame_cx = 320.

  3. Turn errorNormalised steering signal[-1, +1]

    turn_err = (bbox_cx − frame_cx) / 320. Right-of-centre is positive (turn right); left is negative.

  4. Distance errorFrom apparent heighttarget 0.30

    dist_err = (bbox_h / 480) − 0.30. Positive (subject is close) drives backward; negative drives forward.

  5. DriveCompose & emit10 Hz

    speed = −dist_err × 60, turn = turn_err × 40; send M:speed,turn\n over UART.

  6. Loss recoveryRe-acquisition2 s timeout

    If no bbox for 2 s, stop motors and enter PATROL; re-acquire by slow sweep until a person appears.

Queue design · non-blocking inter-thread plumbing
4 queues · bounded
  1. Framesframe_queuemaxsize 2

    Always-latest frame from camera; CVPipeline drops stale frames so the camera thread never blocks behind inference.

  2. Scenesscene_queuemaxsize 2

    Always-latest SceneState; BehaviorManager always reads the freshest analysis, never a backlog.

  3. Commandscommand_queuemaxsize 10

    Buffered drive commands; SerialComm drains at line rate; back-pressure limited to a few hundred ms.

  4. Telemetrytelemetry_queuemaxsize 20

    ESP32 telemetry arrives at 20 Hz; Pi 5 reads at its own pace; old data is auto-dropped at queue boundary.

  5. InvariantProducers never blockdesign rule

    If a queue is full, drop oldest or skip. Stale data is worse than no data — this rule applies everywhere.

  6. EmergencyE: bypasses everything<50 ms

    Emergency stop command E:\n is written to serial immediately in the BehaviorManager thread — no queue, no buffering.

Fall detection & safety interlocks
<50 ms · hardware-backed
  1. WatchTelemetry monitor20 Hz

    SerialComm reads pitch from every T: packet and stores the latest value in shared state.

  2. DetectFall threshold|pitch| > 45°

    BehaviorManager checks: if |pitch| > 45°, the FALL protocol triggers immediately on the next 10 Hz tick.

  3. RecoverControlled sit2 s

    Send P:SIT to ESP32; ESP32 lowers legs through normal interpolation; PID disabled during the descent.

  4. E-STOPGesture cutoff<50 ms

    Open palm detected → skip queue → write E:\n direct to serial → motors cut in < 50 ms end-to-end.

  5. Boot-safeStrapping pin avoidanceat power-up

    GPIO 12 unused. GPIO 14 (ENC1_B) LOW at boot — safe. GPIO 15 (H_IN4) HIGH at boot — killed in first line of setup(): digitalWrite(15, LOW). No net motor glitch; pin map locked in firmware header.

  6. HardwareBMS cutoffcell-level

    Hardware-level: if cell voltage drops below threshold, the BMS cuts power before firmware can react. Never bypassed.
